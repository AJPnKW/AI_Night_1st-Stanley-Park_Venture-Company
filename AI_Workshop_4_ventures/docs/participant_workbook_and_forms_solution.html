Participant Workbook and Forms Solution

Recommended end-state solution for individualized participant workflow using GitHub Pages, Google Forms, Google Sheets, and an optional Apps Script supplement.

What problem this solves

The current local browser workbook is useful for live note-taking, but it does not fully behave like an LMS. If you want each participant to have an individual workflow, select prompt options, record notes, capture choices, preserve progress, and end with a saveable or emailable result, then the solution needs a structured forms layer rather than GitHub Pages alone.

Vision translated into concrete design

What the learner experience should feel likeConcrete implementation meaning
A private workbook that belongs to the learnerEach participant gets a prefilled form link or workbook entry path plus a local browser workbook fallback.
A guided notebook, not a blank pageEach activity starts with a selected category, level, starter card, and sentence-frame help.
A learning process, not only a data formEach section asks what changed, what worked, what gap remained, and what to try next.
Support for fast finishersBranching routes quick finishers to Remix or Stretch follow-up work instead of waiting.
Support for stuck learnersGuided sentence stems and narrower follow-up choices reduce overload and blank-page pressure.

Suggested participant workflow

StepExperience
1. Entry pointParticipant starts from the public site and opens the organizer-issued workbook form link.
2. IdentificationThe form captures participant name or alias, role, group, and pathway assignment.
3. Guided sectionsThe form walks the learner through the session blocks: prompt improvement, image prompt, reflection, and follow-up.
4. Prompt selectionThe learner chooses from starter prompt options or a guided level path.
5. Note captureThe learner records what they changed, what result they got, and what gap remained.
6. Adaptive progressionBranching sends learners to Guided, Build, Remix, or Stretch follow-up items.
7. Completion outputThe response lands in Sheets and can optionally trigger an Apps Script summary email or PDF workflow.

What GitHub Pages should still do

  • Stay as the calm public front door and reference layer.
  • Host the exercise cards, examples, KB, and facilitator resources.
  • Provide printable local workbook pages for backup and offline-ish fallback.
  • Link out to the organizer-issued per-participant form when structured tracking is needed.

What the form should capture

  • Participant identifier or alias
  • Device/account readiness
  • Chosen exercise level
  • Selected starter prompt card
  • Prompt revision notes
  • Expectation versus result
  • Reflection answers
  • Need-help or accommodation flags
  • Consent to receive a summary email if that workflow is enabled

Enhanced operational modelAdd Apps Script to email a summary, generate a PDF, and write a completion status back to Sheets.

Why this is the right answer

This approach is more realistic than trying to force GitHub Pages to act like an LMS. It uses each tool for what it is good at: Pages for public content and local interaction, Forms for structured participant flow, Sheets for visibility, and Apps Script only where actual workflow automation is needed.
